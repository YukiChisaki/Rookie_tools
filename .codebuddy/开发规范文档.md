# 开发规范文档

## 技术栈版本约束

- 严格遵循 `package.json` 中声明的版本范围，禁止未经评估的主版本升级。
- 新增依赖须经确认，避免引入冗余包。

## 目录组织规范

- `src/modules/`：业务功能模块，按功能域划分目录，每个模块自包含页面级逻辑。
- `src/components/`：通用组件。
  - `layout/`：布局级组件。
  - `ui/`：基础 UI 组件。
- `src/stores/`：Pinia 状态管理，按领域拆分文件。
- `src/services/`：数据服务与外部交互（IndexedDB、文件解析等）。
- `src/composables/`：可复用的 Vue 组合式逻辑。
- `src/types/`：全局 TypeScript 类型定义。
- `src/utils/`：纯工具函数，无副作用。
- `src/styles/`：全局样式与主题变量。

## 文件命名规范

- 模块目录：kebab-case。
- Vue 单文件组件：PascalCase.vue。
- TypeScript 文件：camelCase.ts。
- Store 文件：与领域同名，camelCase.ts。

## 模块划分规范

- 每个 `modules/` 子目录为一个独立功能域，内部可包含子组件、局部类型与逻辑。
- 模块间通信优先通过 Pinia Store，避免深层组件传参。

## 状态管理规范

- 按领域拆分 Store，禁止集中式大 Store。
- Store 职责清晰：只管理状态与同步操作，异步逻辑下沉至 services 或 composables。

## 样式规范

- 优先使用 Tailwind CSS 工具类，减少自定义 CSS。
- 全局主题变量统一维护在 `src/styles/`。
- 禁止在组件内写死颜色值，统一使用主题变量或 Tailwind 语义化类名。
- 所有颜色色值全部采用十六进制格式

## 数据管理规范

- IndexedDB 表结构变更须兼容历史数据，避免用户数据丢失。
- 标签数据源来自 `data/lists/` 下的 txt 文件，禁止在代码中硬编码标签内容。
- 文件解析逻辑封装于 services，模块层只调用结果。

## 构建与发布规范

- 使用 Vite 默认构建流程，生产构建前执行类型检查。
- 禁止在生产构建产物中保留调试日志或测试数据。

## 版本管理规范

- 每完成一个功能即提交一次 Git，保持提交粒度清晰。
- 提交信息简明扼要，说明本次提交的核心变更。
- 禁止提交包含调试代码或未完成的工作。
- 功能开发完毕后再合并至主分支，保持主分支稳定。
- 每次合并前，请先在 `.codebuddy/plans/` 下创建计划文档，记录本次变更。
