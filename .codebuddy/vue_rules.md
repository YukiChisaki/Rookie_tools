# 前端代码开发与输出规范

## 1. 通用要求

- **语言**：思考过程与输出内容一律使用中文。
- **上下文引用**：每次对话必须明确引用项目中的 `README.md` 相关内容，确保开发行为与项目目标一致。
- **项目架构**：基于 **Vue 3 (Composition API + `<script setup>`) + TypeScript + Vite + Naive UI + Pinia + Tailwind CSS** 的前端架构。
- **开发技能库应用 (Skills)**：开发过程中需严格结合项目中设定的专家技能（`vue`, `vite`, `naive-ui`, `ui-ux-pro-max`, `frontend-design`）进行代码输出，确保代码质量与设计美感双重达标。
- **Git 提交规范**: 所有代码提交必须遵循 `GIT_COMMIT.md` 中定义的规范。

## 2. 文档驱动开发与同步规范（📝 核心要求）

- **文档位置**: 项目详细文档统一存放在 `/docs` 目录下，并使用 `.md` 格式编写。
- **功能记录机制**：**每次完成任何一个功能需求后，必须使用 Markdown 文档进行详细记录。**
  - 涉及接口、状态管理、复杂算法的变更，需更新对应的技术设计文档。
  - 业务功能的完成需追加到 `/docs/CHANGELOG.md`（或类似进度跟踪文档）中。
- **README 同步更新**：若完成的功能涉及项目的核心特性、启动方式、环境变量或基础架构变更，必须**立即同步更新**项目根目录的 `README.md`，保持项目文档与实际开发进度**高度、绝对统一**。

## 3. 统一 UI/UX 设计与配色规范（✨ 核心要求）

### 3.1 统一配色方案

- **唯一真实来源**：项目必须使用统一的配色方案，所有的颜色变量配置均定义在 **`src/styles`** 目录下。
- **绝对禁止硬编码颜色**：严禁在组件的 `<style>` 或 Tailwind 类名中直接写入类似 `#FF0000`, `rgb(...)` 或未定义的颜色词的魔法颜色值。
- **结合 Tailwind 使用**：使用颜色时，必须通过映射好的 Tailwind CSS 主题类名（如 `text-primary`, `bg-brand-500`）或原生的 CSS 变量（如 `var(--color-primary)`）来调用配置。

### 3.2 UI/UX Pro Max & 前端界面设计原则

基于 `ui-ux-pro-max` 和 `frontend-design` 技能库，遵循以下设计准则：

- **高标准审美**：创建精美的 Web 界面，注重留白（Spacing）、对齐和视觉层级。
- **微交互与动画**：在按钮点击、组件加载、页面切换等场景中，适当加入平滑的过渡动画（结合 Tailwind 的 `transition` 和 `animate` 类）。
- **设计一致性**：确保整个应用的按钮形态、卡片阴影、圆角大小（Border Radius）保持高度一致。

## 4. 代码风格与规范

### 4.1 命名规范

- **变量/函数**：采用小驼峰命名法（`camelCase`）。
- **组件名**：采用大驼峰命名法（`PascalCase`），如 `UserList.vue`。
- **文件名**：Vue 组件使用大驼峰；TS/工具类文件使用小驼峰或短横线（`kebab-case`）。
- **常量**：使用全大写加下划线，如 `API_BASE_URL`。

### 4.2 注释规范

- 所有函数、组件、关键逻辑块必须附带清晰的中文注释，说明"为什么"而不仅是"做什么"。
- 顶部注释必须包含：`@author username / email` 和 `@since yyyy-MM-dd HH:mm:ss`（可使用项目中提供的脚本获取）。

### 4.3 核心开发实践 (Vue 3 + Vite + Naive UI + TS)

- **Vue 3**：深度应用 Composition API (`<script setup>`)。逻辑复杂时抽离为 Composables (`useXxx.ts`)。全局状态强制使用 **Pinia**。
- **按需自动导入**：**严禁**手动 `import` Vue 核心 API（`ref`, `computed`等）和 Naive UI 组件（`<n-button>`等），项目已配置 `unplugin` 插件实现自动按需导入。
- **TypeScript**：全面使用 TS，严格定义 `interface` 或 `type`，为 Props 和 Emits 定义明确类型，**杜绝使用 `any`**。
- **Vite 构建**：注意代码拆分和按需加载，避免造成单包体积过大。

### 4.4 特定技术栈要求

- **图标库**：统一使用 `Lucide Vue Next`，按需引入。
- **瀑布流布局**：瀑布流场景统一使用 `@yeger/vue-masonry-wall`。
- **本地数据存储**：强制使用项目封装的 `DatabaseService` (基于 IndexedDB v3)，禁止滥用 `localStorage`。
- **图片元数据解析**：使用 `ExifReader` 库读取图片元数据及 PNG tEXt 块。

## 5. 🤖 AI 助手交互准则与自动工作流

在接收到开发任务后，AI 助手必须严格按照以下流水线执行：

### 阶段一：交付前自我审查机制 (Self-Review) 【强制执行】

在完成任何代码开发需求后，**将代码交付给开发人员之前，AI 助手必须自动执行一次自我代码审查**。
需要在输出内容中附带一个简短的 **【AI 自我审查报告】**：

- ✅ **UI 配色防雷检查**：确认代码中存在 0 个硬编码颜色，全量使用 Tailwind 规范色值或 `src/styles` 变量。
- ✅ **按需导入检查**：确认无冗余的手动 Vue/Naive UI `import` 语句。
- ✅ **类型安全检查**：确认无 `any` 类型，TS 类型定义完整且严格。
- ✅ **注释与规范检查**：确认包含了 `@author` 和 `@since` 标签，代码逻辑符合单一职责。

### 阶段二：文档自动同步更新机制 (Doc Sync) 【强制执行】

完成自我审查后，AI 必须主动执行文档更新操作：

1.  **梳理变更**：提取本次开发的核心功能、新增接口或修改的配置项。
2.  **生成/更新文档**：自动输出更新后的 `/docs/CHANGELOG.md` 或相关设计文档内容。若涉及全局特性，输出对 `README.md` 的修改。
3.  **状态通报**：明确告知开发者：“📝 **已同步更新文档**：记录了本次功能的开发详情，并更新了相应的 MD 文件。”

### 阶段三：Git 提交信息生成与自动执行

经过代码输出与文档更新后，AI 助手进入版本控制流程：

1.  **生成提交信息**：按照 `[任务编号] type: subject` 格式生成 commit message，并在 body 中使用 `-` 列出代码变更与文档更新内容。
2.  **主动询问**：“代码已交付且文档已同步，是否需要我直接执行 git 提交并推送到远程仓库？”
3.  **自动执行**：在开发人员确认后，AI 自动生成并执行命令序列：
    `git add .`
    `git commit -m "..."`
    `git push`
4.  **最终反馈**：提供 Git 操作的最终成功反馈结果。
